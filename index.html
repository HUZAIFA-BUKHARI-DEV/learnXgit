⎈ git cheatsheet · dev + plain english

technical purpose + plain translation for junior devs — same professional theme

core 1. BASIC · daily operations

−
git initinitialize repository
technical
Creates a new .git subdirectory, enabling version control in an existing project. Transforms a regular folder into a git repository.
plain english 🧠 tells your folder: "start recording every change i make from now on". creates a hidden storage for future snapshots.
example / real use
$ cd ~/projects/new-api
$ git init
# now you can track files
git clone <url>copy remote repo
technical
Clones a remote repository (all commits, branches) into a new directory. Standard way to get a working copy from GitHub/GitLab.
plain english 📦 downloads a complete project from the internet (including all history) to your computer.
example
$ git clone https://github.com/org/service.git
# entire repo lands in ./service
git statusshow working tree state
technical
Displays which changes are staged, unstaged, or untracked. First command to run when you need orientation.
plain english 🔍 asks git: "what's different since the last snapshot? what's ready to save?"
example
$ git status
# On branch main, modified: index.html, untracked: .env
git add <path> / -pstage changes
technical
Stages modifications. -p allows interactive hunk staging (partial commits). Moves changes from working dir to index.
plain english ➕ selects which changed files (or parts of files) you want to include in the next snapshot.
example
$ git add src/*.js
$ git add -p # choose pieces interactively
git commit -mrecord snapshot
technical
Takes staged changes and creates a new commit with a message. Permanent entry in project log.
plain english 📸 saves a permanent picture of the staged changes, with a short label describing what you did.
example
$ git commit -m "feat: add rate limiting middleware"
git log (--oneline)show history
technical
Displays commit history; --oneline compresses each commit to one line (hash + title). Useful for quick overview.
plain english 📋 shows the album of all past snapshots, with dates and messages.
example
$ git log --oneline -5
# last 5 commits, one line each
git diff [--staged]show line changes
technical
Without flags: diff working dir vs index. --staged: diff index vs last commit (what would be committed).
plain english 📏 compares current unsaved changes against the last snapshot. helps you review before staging.
example
$ git diff # unstaged changes
$ git diff --staged # staged changes (review before commit)

branching 2. MEDIUM · collaboration & branches

−
git branchlist / create branches
technical
List existing branches. With a name, creates a new branch (but does not switch).
plain english 🌿 shows all parallel versions of your project, or makes a new experimental line.
example
$ git branch feat/auth
$ git branch --all
git switch <branch>move HEAD
technical
Switch to another branch (updates working tree). -c creates and switches.
plain english 🔄 jumps between different parallel versions of your code.
example
$ git switch -c feature/pagination
$ git switch main
git merge <branch>integrate changes
technical
Merges specified branch into current branch. May produce merge commit or fast-forward.
plain english 🔀 combines work from a different branch into your current one.
example
$ git checkout main
$ git merge feature/pagination
git pull / pushsync remote
technical
Pull = fetch + merge remote changes. Push = upload local commits to remote.
plain english ⬇️ pulls updates from github; ⬆️ sends your snapshots to github.
example
$ git pull origin main
$ git push origin feature/x
merge conflictsmanual resolution
technical
When automatic merge fails, git marks conflicted files. Edit, remove markers, stage and commit.
plain english ⚠️ git is confused because two different changes touched the same line. you must manually pick the correct version.
example
$ git status
# edit file, remove <<< === >>>
$ git add .
$ git commit
git stash / popshelve changes
technical
Stash saves uncommitted work and cleans working dir. pop reapplies last stash.
plain english 🧹 temporarily puts aside unfinished work so you can do something else, then brings it back.
example
$ git stash
$ git switch hotfix
$ git stash pop

history 3. ADVANCED · rewriting & debugging

−
git rebasereapply commits
technical
Takes commits from current branch and replays them onto target. Results in linear history. Avoid on shared branches.
plain english ⏳ rewrites history so it looks like your work started from a newer point – keeps timeline clean.
example
$ git rebase main
$ git rebase --continue
git rebase -i HEAD~Ninteractive rewrite
technical
Opens editor to squash, reword, drop, or reorder last N commits before push.
plain english ✏️ lets you clean up messy commit history (combine, rename, delete) before sharing.
example
$ git rebase -i HEAD~4 # reorganise last 4 commits
git reflogreference log
technical
Shows history of where HEAD pointed. Essential to recover lost commits after reset / rebase.
plain english 🆘 a diary of everything you did – even if you deleted something, you can find it here.
example
$ git reflog
$ git reset --hard HEAD@{2}
git cherry-pick <hash>apply specific commit
technical
Copies a commit from any branch into current HEAD. Useful for backporting fixes.
plain english 🍒 grabs a single change from another branch and applies it to your current one.
example
$ git cherry-pick f3a1d4b
git bisectbinary search
technical
Binary search through commits to locate where a bug was introduced. Marks commits as good/bad.
plain english 🔎 helps you find which exact commit caused a bug by jumping through history and asking "is this version good or bad?"
example
$ git bisect start
$ git bisect bad HEAD
$ git bisect good v1.2.3
# then test & mark good/bad
